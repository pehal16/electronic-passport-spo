# Роли и права доступа

Проект: `Электронный паспорт группы СПО`

## 1. Общий принцип доступа

Система строится по модели `RBAC`: у каждого пользователя есть роль, и каждая роль видит только тот набор данных и действий, который ей необходим.

Главный принцип:
1. одна база данных;
2. разные уровни доступа;
3. разные интерфейсы для разных ролей;
4. ограничение видимости персональных данных.

## 2. Основные роли

1. Администратор
2. Куратор группы
3. Преподаватель
4. Студент
5. Родитель / законный представитель
6. Администрация колледжа

## 4. Что должна видеть каждая роль

## 4.1 Администратор

### Видит
1. все группы;
2. всех пользователей;
3. все роли и связи;
4. справочники;
5. документы;
6. системные настройки;
7. полную аналитику.

### Может
1. создавать пользователей;
2. назначать роли;
3. настраивать права;
4. привязывать преподавателей к группам и дисциплинам;
5. привязывать родителей к студентам;
6. редактировать структуру системы;
7. архивировать и восстанавливать данные.

## 4.2 Куратор

### Видит
1. полную карточку своей группы;
2. карточки всех студентов своей группы;
3. посещаемость по всем дисциплинам;
4. сводные оценки;
5. замечания и похвалу;
6. документы группы;
7. сведения о родителях;
8. риски и предупреждения.

### Может
1. редактировать паспорт группы;
2. вести заметки по студентам;
3. добавлять записи о беседах;
4. публиковать объявления группе;
5. отправлять уведомления родителям;
6. видеть сводные отчёты по группе.

### Не может
1. менять системные роли;
2. редактировать оценки за преподавателя;
3. видеть чужие группы.

## 4.3 Преподаватель

### Видит
1. только свои дисциплины;
2. только те группы, где он ведёт занятия;
3. учебную информацию по студентам;
4. свои журналы и ведомости.

### Может
1. выставлять оценки;
2. отмечать посещаемость на своих занятиях;
3. оставлять комментарии;
4. добавлять учебные материалы;
5. публиковать учебные объявления;
6. формировать ведомость по предмету.

### Не может
1. видеть внутренние заметки куратора;
2. редактировать данные других преподавателей;
3. видеть полные персональные данные всех студентов.

## 4.4 Студент

### Видит
1. свой профиль;
2. свою группу;
3. своё расписание;
4. свои оценки;
5. свою посещаемость;
6. свои задолженности;
7. объявления;
8. задания;
9. свои достижения.

### Может
1. просматривать свои данные;
2. подтверждать ознакомление;
3. загружать задания, если функция включена;
4. скачивать материалы.

### Не может
1. видеть данные других студентов;
2. редактировать журнал;
3. видеть служебные заметки.

## 4.5 Родитель

### Видит
1. профиль своего ребёнка в ограниченном виде;
2. оценки;
3. посещаемость;
4. пропуски;
5. долги;
6. важные объявления;
7. сообщения куратора;
8. приглашения на собрания.

### Может
1. просматривать данные;
2. подтверждать ознакомление;
3. отправлять короткое сообщение куратору через форму обратной связи.

### Не может
1. видеть других студентов;
2. видеть внутренние служебные заметки;
3. редактировать данные.

## 4.6 Администрация колледжа

### Видит
1. сводные данные по группам;
2. отчёты по посещаемости;
3. отчёты по успеваемости;
4. риски по студентам и группам;
5. качество заполнения журналов.

### Может
1. просматривать аналитику;
2. выгружать отчёты;
3. контролировать полноту данных.

### Не может
1. менять системную архитектуру без прав администратора.

## 5. Логика ограничений

## 5.1 Ограничения по студенту

1. студент видит только свою карточку;
2. родитель видит только карточку своего ребёнка;
3. преподаватель видит только учебный блок карточки студента;
4. куратор видит расширенную карточку только своей группы.

## 5.2 Ограничения по группе

1. куратор работает только со своей группой;
2. преподаватель работает только с закреплёнными за ним группами;
3. администрация видит все группы в режиме чтения и аналитики.

## 5.3 Ограничения по персональным данным

1. домашний адрес, паспортные и медицинские сведения не доступны всем ролям;
2. чувствительные данные доступны только ролям по политике колледжа;
3. доступ к чувствительным полям логируется в аудит.

## 6. Рекомендуемые уровни доступа в MVP

Для первой версии MVP:
1. `admin`
2. `curator`
3. `teacher`
4. `student`

Роли `parent` и `management` внедряются этапом 2, но закладываются в архитектуру сразу.

## 8. Техническая реализация RBAC (рекомендация)

1. Проверка роли в JWT + проверка контекста (`group_id`, `discipline_id`, `student_id`).
2. Политики доступа на уровне сервисов backend, а не только UI.
3. Полевая фильтрация чувствительных атрибутов в сериализаторах.
4. Аудит критичных операций:
   - выставление/изменение оценки,
   - изменение посещаемости,
   - просмотр чувствительных данных,
   - изменение ролей и привязок.

## 9. Итог

Система реализуется как единая экосистема с роль-ориентированными интерфейсами.
Ключевое правило: каждая роль видит только то, что ей нужно для работы, учёбы или контроля, и не больше.
