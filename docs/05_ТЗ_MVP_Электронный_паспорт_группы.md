# ТЗ на MVP: «Электронный паспорт группы СПО»

## 1. Цель MVP

Создать рабочую цифровую систему для куратора, преподавателя, студента, родителя и администратора, объединяющую:
1. паспорт группы,
2. карточки студентов,
3. журнал оценок,
4. посещаемость,
5. уведомления,
6. базовую аналитику риска.

## 2. Технологический стек

1. Frontend: `Next.js` + `TypeScript`
2. Backend: `FastAPI`
3. DB: `PostgreSQL`
4. Миграции: `Alembic`
5. Auth: JWT (`access token`)
6. Развертывание: локально + `docker-compose` (когда Docker доступен)

## 3. Обязательный функционал MVP

1. Авторизация и ролевой доступ:
   - MVP-1: `admin/curator/teacher/student`
   - Архитектурно заложить: `parent/management` (для этапа 2)
2. Паспорт группы (базовые вкладки)
3. Список студентов и карточка студента
4. Журнал оценок по урокам
5. Модуль посещаемости
6. Уведомления и подтверждение ознакомления (для ролей MVP-1)
7. Ограниченный кабинет студента
8. Базовый риск-скоринг по пропускам и долгам

## 4. Границы MVP

В MVP не включаем:
1. мобильное приложение,
2. сложный документооборот,
3. расширенную интеграцию с внешним расписанием,
4. сложные BI-отчеты,
5. расширенный модуль практики,
6. полный кабинет родителя,
7. полный кабинет администрации колледжа (management).

## 5. Нефункциональные требования

1. Русский язык интерфейса.
2. Крупные кнопки и простой UI.
3. 3-5 кликов до частых действий.
4. Производительность: основные страницы до 2-3 секунд.
5. Логирование ключевых действий.
6. Изоляция данных по ролям.
7. Защита персональных данных (минимизация видимости).

## 6. API-контуры (укрупненно)

1. `/auth/*` — вход, профиль
2. `/users/*`, `/roles/*` — управление доступом
3. `/groups/*`, `/students/*`, `/disciplines/*` — справочники
4. `/passports/*` — паспорт группы
5. `/lessons/*`, `/grades/*`, `/attendance/*` — учебный контур
6. `/notifications/*` — уведомления
7. `/risks/*` — сигналы риска
8. `/reports/*` — базовые отчеты

## 7. Критерии приемки MVP

1. Каждая роль заходит под своей учетной записью и видит только свой контур.
2. Преподаватель может зафиксировать посещаемость и оценки по занятию.
3. Куратор видит сводку по группе и карточки студентов.
4. Студент и родитель видят ограниченный кабинет без чужих данных.
5. Уведомления отправляются и подтверждаются.
6. Риски формируются автоматически хотя бы по двум правилам:
   - N пропусков за период,
   - M долгов по дисциплинам.

## 8. План реализации (рекомендуемый)

1. Спринт 1: роли, доступы, справочники, группы/студенты.
2. Спринт 2: паспорт группы + карточка студента.
3. Спринт 3: журнал оценок + посещаемость + уроки.
4. Спринт 4: кабинет студента + уведомления + risk-engine (база).
5. Спринт 5: полировка UX + приемка MVP-1.
6. Этап 2: родитель и management, расширенные отчеты.

## 9. Что уже есть в текущем каркасе

2. Базовые справочники.
3. Шаблоны и занятия.
4. Оценивание.
5. Журнал.

1. Новые роли (`curator/student/parent/management`) и матрицу доступа.
2. Сущности родителя и привязку к студенту.
3. Паспорт группы как центральный агрегатор.
4. Отдельный модуль посещаемости с причинами и порогами.
5. Ограниченные кабинеты студента и родителя.
6. Уведомления с подтверждением ознакомления.
7. Базовый risk-engine.
